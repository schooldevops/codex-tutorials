# Codex 사용 가이드

> 목적: Codex를 이용해 특정 애플리케이션을 기획부터 개발, 검증, 배포까지 단계적으로 진행하기 위한 실무형 가이드

## 0. 시작 전 원칙
- Codex는 “요청을 정확히 정의할수록” 결과 품질이 올라갑니다.
- 한 번에 큰 작업보다, 작은 단위로 나눠 반복하는 방식이 안전합니다.
- 각 단계마다 테스트/검증을 넣어야 회귀(regression)를 줄일 수 있습니다.

## 1. 초기 세팅

### 1-1. 저장소 기본 파일 준비
- `AGENTS.md`: Codex 작업 규칙 (이미 생성)
- `README.md`: 프로젝트 소개/실행 방법
- `.gitignore`: 빌드 산출물, 환경파일 제외
- `docs/`: 요구사항/설계 문서 보관 폴더

### 1-2. 프로젝트 목표 문장 만들기
아래 항목을 1~2문장으로 먼저 확정합니다.
- 무엇을 만드는가? (예: 팀용 할 일 관리 웹앱)
- 누가 사용하는가? (예: 5~20명 규모 스타트업 팀)
- 성공 기준은 무엇인가? (예: 2주 내 MVP 배포, 주요 기능 3개 완성)

### 1-3. Codex에게 첫 요청 템플릿
```text
이 저장소에서 [앱 이름] MVP를 개발하려고 한다.
기술스택은 [예: Next.js + TypeScript + PostgreSQL]이다.
아래 순서로 진행해줘:
1) 현재 구조 점검
2) 초기 폴더/설정 파일 생성
3) 실행 및 테스트 스크립트 구성
각 단계 후 변경 파일과 검증 결과를 요약해줘.
```

## 2. 개발 프로세스 설계 (MVP 기준)

### 2-1. 기능을 수직 슬라이스로 분해
예시:
1. 인증(로그인/로그아웃)
2. 핵심 도메인 CRUD
3. 목록/검색/필터
4. 권한 처리
5. 기본 대시보드

각 기능을 아래 4개 단위로 고정해 작업합니다.
- API/비즈니스 로직
- 데이터 모델/마이그레이션
- UI/UX
- 테스트(단위/통합)

### 2-2. 이슈 단위 정의 규칙
- 하나의 이슈는 “반나절~1일 내 완료 가능한 크기”로 제한
- 완료 조건(DoD)을 체크리스트로 명시
- 테스트 케이스를 이슈 생성 시 같이 정의

## 3. Codex 단계별 활용 방법

### 3-1. 탐색 단계 (코드베이스 이해)
요청 예시:
```text
코드베이스를 탐색해서 아래를 요약해줘.
- 디렉토리 구조
- 실행/빌드/테스트 방법
- 핵심 진입점 파일
- 당장 구현 가능한 첫 기능 제안 3개
```

기대 산출물:
- 구조 요약
- 진입 명령어 목록
- 우선순위 기능 제안

### 3-2. 설계 단계 (구현 전 정렬)
요청 예시:
```text
[기능명] 구현 전에 설계를 먼저 작성해줘.
포함 항목:
1) 데이터 모델 변경
2) API 스펙(요청/응답)
3) UI 상태(로딩/에러/빈 상태)
4) 테스트 전략
설계 확정 전에는 코드 수정하지 말아줘.
```

기대 산출물:
- 구현 계획
- 리스크/의존성
- 테스트 전략

### 3-3. 구현 단계 (작은 단위 반복)
요청 예시:
```text
위 설계대로 1단계(API)만 구현해줘.
다음 규칙을 지켜줘:
- 최소 수정
- 변경 파일 목록 제시
- 테스트 명령 실행 후 결과 보고
UI 작업은 아직 하지마.
```

핵심 포인트:
- “이번 턴에 할 범위/하지 않을 범위”를 명확히 지시
- 단계 완료 후 반드시 테스트 결과 확인

### 3-4. 검증 단계 (품질 고정)
요청 예시:
```text
지금 변경 기준으로
1) 실패 가능성이 높은 경계 케이스 5개
2) 누락된 테스트
3) 리팩터링이 필요한 부분
을 우선순위로 리뷰해줘.
```

기대 산출물:
- 버그 위험 목록
- 추가 테스트 항목
- 기술부채 정리 포인트

## 4. 초기 세팅부터 배포까지 상세 단계

### Step 1) 기술스택 확정
- 프론트: Next.js / React / Vue 중 선택
- 백엔드: Node / Python / Go 중 선택
- DB: PostgreSQL / MySQL / SQLite 중 선택
- 배포: Vercel / AWS / Render 중 선택

산출물:
- `docs/stack.md`

### Step 2) 개발환경 표준화
- Node/Python 버전 고정
- 패키지 매니저 고정(`npm`, `pnpm`, `yarn` 중 1개)
- `.env.example` 작성

산출물:
- 버전 파일, 환경 변수 예시 파일

### Step 3) 프로젝트 스캐폴딩
- 기본 디렉토리 생성
- 린트/포맷터/테스트 러너 설치
- 실행 스크립트 정의(`dev`, `build`, `test`, `lint`)

산출물:
- 실행 가능한 최소 앱 구조

### Step 4) 도메인 모델 정의
- 핵심 엔티티/관계 정리
- 스키마 및 마이그레이션 작성
- 시드 데이터 준비

산출물:
- DB 모델 + 마이그레이션 파일

### Step 5) 핵심 기능 1개 완성
- 로그인 또는 핵심 CRUD 중 하나 먼저 완료
- API, UI, 테스트를 한 세트로 구현

산출물:
- E2E 기준으로 동작 확인 가능한 첫 기능

### Step 6) 관측성/안정성 보강
- 에러 처리 정책 통일
- 로깅/모니터링 기본 연결
- 공통 예외 응답 스키마 정리

산출물:
- 운영 중 디버깅 가능한 최소 장치

### Step 7) 배포 파이프라인 연결
- 브랜치 전략 확정
- CI에서 lint/test/build 자동화
- 스테이징 배포 후 체크리스트 검증

산출물:
- 자동 검증 + 배포 가능한 상태

## 5. Codex 프롬프트 운영 규칙

### 좋은 요청 패턴
- 목적 + 범위 + 제약 + 완료조건을 함께 명시
- “한 번에 하나의 단계”만 실행 요청
- 결과 보고 형식을 지정

예시:
```text
[기능명]의 DB 마이그레이션만 작성해줘.
제약:
- 기존 데이터 호환성 유지
- 롤백 가능해야 함
완료 조건:
- 마이그레이션 파일 생성
- 적용/롤백 명령 안내
- 영향 받는 코드 경로 요약
```

### 피해야 할 요청 패턴
- “전체 다 만들어줘” 같은 범위 불명확 요청
- 테스트/검증 없는 즉시 구현 요청
- 스택/버전 미정 상태에서 구현 시작

## 6. 실전 운영 루틴 (권장)
1. 아침: 오늘 구현할 이슈 1~2개 확정
2. 구현 전: Codex에게 설계안 먼저 요청
3. 구현: 단계별로 코드 생성/수정
4. 검증: 테스트 + 코드리뷰 관점 점검
5. 마감: 변경 요약/리스크/다음 작업 기록

## 7. 완료 보고 템플릿
아래 포맷으로 Codex 응답을 요청하면 관리가 쉬워집니다.

```text
아래 형식으로 결과를 보고해줘:
1) 변경 파일
2) 핵심 변경 사항
3) 실행한 테스트/린트 명령과 결과
4) 남은 리스크
5) 다음 단계 제안 (최대 3개)
```

## 8. 추천 다음 작업

- 확정한 앱 유형 기준으로 `docs/requirements.md` 작성

## 9. Git Worktree 기반 안전 개발 가이드

> 목표: 기능별로 격리된 작업 디렉토리에서 안전하게 개발하고, 검증 완료 후 작업 브랜치에 머지

### 9-1. 왜 Worktree를 쓰는가
- 기능 단위로 작업 디렉토리를 분리해 충돌을 줄일 수 있습니다.
- 같은 저장소에서 여러 브랜치를 동시에 열어 비교/검증할 수 있습니다.
- 실수로 다른 기능 변경을 섞어 커밋하는 위험이 줄어듭니다.

### 9-2. 브랜치 모델 (권장)
- 기준 브랜치: `main` (또는 팀의 기본 통합 브랜치)
- 작업 브랜치: `feature/<기능명>`
- 핫픽스 브랜치: `hotfix/<이슈명>`

규칙:
- 기능 1개당 브랜치 1개
- 브랜치 생성 직후 Worktree 생성
- Worktree별 `.env`/실행 포트 분리

### 9-3. 최초 준비 명령
```bash
git fetch --all --prune
git switch main
git pull origin main
git worktree list
```

### 9-4. 기능 작업용 Worktree 생성
예시: `feature/auth-login` 기능 작업
```bash
# 저장소 루트 기준 상위에 worktrees 폴더를 두는 방식 (권장)
mkdir -p ../worktrees
git worktree add ../worktrees/auth-login -b feature/auth-login main
```

생성 후 작업:
```bash
cd ../worktrees/auth-login
git status -sb
```

### 9-5. Codex를 Worktree에서 안전하게 사용하는 방법
각 단계별로 Codex에게 명확히 지시합니다.

요청 예시:
```text
현재 브랜치(feature/auth-login)에서만 작업해줘.
요구사항 범위는 로그인 API + 로그인 폼 UI까지다.
아래 규칙을 지켜줘:
1) 최소 수정
2) 변경 파일 목록 보고
3) 테스트/린트 실행 결과 보고
4) 커밋은 하지 않음
```

권장 운영:
1. 설계 요청
2. API 구현 요청
3. UI 구현 요청
4. 테스트 보강 요청
5. 코드리뷰(리스크 점검) 요청

### 9-6. 동기화(최신 main 반영) 절차
작업 중 `main`이 업데이트되면 주기적으로 반영합니다.

```bash
git fetch origin
git rebase origin/main
```

충돌 시:
1. 충돌 파일 확인
2. 수동 해결
3. 테스트 재실행
4. `git rebase --continue`

주의:
- 복잡한 충돌이 예상되면 작은 단위 커밋 후 리베이스 수행
- 리베이스 후 반드시 회귀 테스트 실행

### 9-7. 머지 전 품질 게이트 체크리스트
- [ ] 요구사항 범위 외 변경 없음
- [ ] lint/typecheck/test 통과
- [ ] 마이그레이션/환경변수 변경 문서화
- [ ] 롤백 가능 여부 확인
- [ ] 리뷰 코멘트 반영 완료

### 9-8. 작업 브랜치로 머지하는 안전 절차
팀 규칙에 따라 `main` 또는 통합 브랜치로 진행합니다.

```bash
# worktree에서 작업 완료 후
git status
git add .
git commit -m "feat(auth): add login api and ui"
git push -u origin feature/auth-login
```

이후 PR 머지(권장: Squash Merge):
1. CI 통과 확인
2. 리뷰 승인
3. Squash Merge
4. main 동기화

로컬 정리:
```bash
cd <repo-root>
git worktree remove ../worktrees/auth-login
git branch -d feature/auth-login
```

원격 브랜치 정리(선택):
```bash
git push origin --delete feature/auth-login
```

### 9-9. 병렬 기능 개발 패턴
여러 기능을 동시에 진행할 때:
- `../worktrees/feature-a`
- `../worktrees/feature-b`
- `../worktrees/feature-c`

각 Worktree에서 포트/환경 파일 분리 예시:
- feature-a: `PORT=3001`
- feature-b: `PORT=3002`
- feature-c: `PORT=3003`

### 9-10. 복구/안전 장치
- 실수로 변경이 꼬이면 `git reflog`로 복구 지점 확인
- 커밋 전 임시 저장은 `git stash -u` 활용
- 대규모 변경 전 태그 생성 권장

예시:
```bash
git tag backup/pre-auth-refactor-2026-02-27
```

### 9-11. Worktree 기반 Codex 요청 템플릿
```text
지금 경로가 worktree(브랜치: feature/<name>)인지 먼저 확인해줘.
확인 후 아래 순서로 진행해줘:
1) 요구사항 재정의 (이번 턴 범위/비범위)
2) 코드 수정 (최소 변경)
3) 테스트/린트 실행
4) 결과 보고 (변경 파일, 테스트 결과, 남은 리스크)
머지나 브랜치 삭제 명령은 내가 요청할 때만 실행해줘.
```

### 9-12. 자동화 스크립트 사용 (`scripts/git-safe-flow.sh`)
반복되는 Worktree 작업을 스크립트로 표준화할 수 있습니다.

```bash
# 도움말
./scripts/git-safe-flow.sh --help

# 1) 기능 Worktree 생성 (feature/auth-login + ../worktrees/auth-login)
./scripts/git-safe-flow.sh create auth-login

# 2) Worktree 목록 확인
./scripts/git-safe-flow.sh list

# 3) 현재 브랜치를 origin/main 기준으로 rebase 동기화
./scripts/git-safe-flow.sh sync main

# 4) 작업 완료 후 Worktree 제거 + 로컬 브랜치 삭제
./scripts/git-safe-flow.sh remove auth-login --delete-branch
```

운영 규칙:
- `sync` 전에 작업 트리가 깨끗한지 확인(커밋 또는 스태시)
- `remove` 전에 변경사항이 남아 있지 않은지 확인
- 원격 브랜치 삭제는 PR 머지 후 명시적으로 수행
