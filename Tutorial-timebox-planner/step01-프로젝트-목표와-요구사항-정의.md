# Step 01 - 프로젝트 목표와 요구사항 정의

## Step 목표
- Time-Box Planner의 MVP 범위를 명확히 정의한다.
- 이후 OpenAPI 명세 작성에 필요한 기능/도메인 요구사항을 확정한다.
- 문서 산출물을 만들어 다음 step에서 바로 구현으로 이어지게 한다.

## 이번 Step 완료 기준 (DoD)
- [ ] 프로젝트 한 줄 목표가 작성되었다.
- [ ] 사용자/사용 시나리오가 정의되었다.
- [ ] MVP 기능 목록과 비기능 요구사항이 확정되었다.
- [ ] 도메인 개념(시간블록, 태그, 상태 등) 초안이 정리되었다.
- [ ] API 계약 초안에 필요한 엔드포인트 후보가 정리되었다.
- [ ] `docs/requirements.md` 파일이 생성되었다.

## 사전 준비
- [ ] 저장소 루트 경로 확인
- [ ] 튜토리얼 작업 브랜치에서 진행 중인지 확인
- [ ] 문서 작성용 에디터 준비

실행 명령:
```bash
pwd
git status -sb
mkdir -p docs
```

## 1) 제품 목표 정의

### 1-1. 제품 한 줄 정의 작성
아래 문장을 채워 작성한다.

```text
Time-Box Planner는 [누구]가 [문제]를 해결하도록 [핵심가치]를 제공하는 서비스다.
```

체크:
- [ ] 누구(타겟 사용자)가 명확하다.
- [ ] 어떤 문제를 해결하는지 측정 가능하게 표현했다.
- [ ] 핵심 가치가 기능이 아닌 결과 중심으로 작성되었다.

### 1-2. 성공 기준 정의
아래 3가지를 숫자로 정의한다.
- 활성 사용자 목표
- 시간블록 생성 완료율 목표
- 주간 재방문율 목표

체크:
- [ ] 각 지표에 기준 기간(예: 2주, 1개월)을 적었다.
- [ ] MVP 종료 시 측정 가능한 형태다.

## 2) 사용자와 시나리오 정의

### 2-1. 핵심 사용자 2종 정의
아래 형식으로 2개 작성한다.

```text
사용자 유형:
- 이름:
- 상황:
- 주요 목표:
- 불편:
```

체크:
- [ ] 사용자별 목표가 서로 다르다.
- [ ] 실제 행동으로 검증 가능한 목표다.

### 2-2. 핵심 사용 시나리오 3개 정의
권장 시나리오:
1. 오늘 일정 시간블록 생성
2. 시간블록 수정/재배치
3. 하루 회고(완료/미완료 확인)

체크:
- [ ] 시작 상태와 종료 상태가 명확하다.
- [ ] 시나리오별 성공 조건이 있다.

## 3) MVP 범위 정의

### 3-1. 반드시 포함할 기능 (In Scope)
예시:
- 시간블록 CRUD
- 날짜별 조회
- 상태 변경(예정/진행/완료)
- 기본 통계(완료 개수, 집중 시간)

체크:
- [ ] 각 기능이 1~2문장으로 설명되어 있다.
- [ ] OpenAPI 엔드포인트로 분해 가능한 수준이다.

### 3-2. 이번 단계에서 제외할 기능 (Out of Scope)
예시:
- 팀 협업/공유
- 실시간 동기화
- 외부 캘린더 양방향 연동

체크:
- [ ] 제외 이유가 함께 적혀 있다.
- [ ] MVP 범위가 과도하게 커지지 않는다.

## 4) 도메인 요구사항 초안

### 4-1. 핵심 엔티티 정의
최소 엔티티 초안:
- User
- TimeBox
- Tag

TimeBox 필드 초안:
- id
- userId
- title
- description
- startAt
- endAt
- status
- priority
- createdAt
- updatedAt

체크:
- [ ] 시간 관련 필드의 타임존 기준을 명시했다.
- [ ] 상태(status) 값 후보를 정했다.
- [ ] nullable/non-nullable을 대략 구분했다.

### 4-2. 비즈니스 규칙 정의
예시:
- `startAt < endAt` 이어야 한다.
- 한 사용자의 동일 시간대 중복 블록 정책을 정의한다(허용/비허용).
- 완료 상태로 변경 시 완료 시각 기록 정책을 정의한다.

체크:
- [ ] 규칙마다 실패 시 에러 메시지 방향이 있다.
- [ ] API 유효성 검사 규칙으로 옮길 수 있다.

## 5) API 계약 준비 체크 (OpenAPI 입력 정보)

다음 step에서 OpenAPI를 작성하기 위해 아래를 미리 정리한다.

### 5-1. 엔드포인트 후보
- `POST /time-boxes`
- `GET /time-boxes`
- `GET /time-boxes/{id}`
- `PATCH /time-boxes/{id}`
- `DELETE /time-boxes/{id}`

체크:
- [ ] 각 엔드포인트의 목적이 1문장으로 정의되었다.
- [ ] 목록 조회 필터(날짜, 상태, 태그) 후보를 정리했다.

### 5-2. 요청/응답 데이터 초안
작성 항목:
- 생성 요청 필드
- 수정 요청 필드
- 단건 응답 필드
- 목록 응답 구조
- 공통 에러 응답 형식

체크:
- [ ] 프런트에서 바로 타입 생성 가능한 구조다.
- [ ] 백엔드 DTO로 변환 가능한 구조다.

## 6) 비기능 요구사항 정의
- 인증: MVP에서 단일 사용자 모드 또는 간단 인증 중 선택
- 성능: 목록 조회 응답 시간 목표 정의(예: 500ms 이내)
- 안정성: 에러 코드 표준화
- 관측성: 최소 로깅 항목 정의(requestId, userId, endpoint)

체크:
- [ ] 비기능 요구사항이 구현/테스트 가능한 문장이다.
- [ ] 운영 단계에서 필요한 최소 기준이 있다.

## 트러블슈팅
- 문제: 요구사항이 너무 커짐
- 해결: Out of Scope를 먼저 확정하고 MVP 기능을 3~5개로 제한

- 문제: API 구조가 자주 바뀜
- 해결: 시나리오 기준으로 엔드포인트 목적부터 고정 후 필드 조정

- 문제: DB 모델이 모호함
- 해결: 엔티티 간 관계(1:N, N:M)와 제약조건을 문장으로 먼저 확정
