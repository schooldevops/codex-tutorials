# Codex 기본 작업 규칙 (Project Root)

> 이 파일은 Codex가 이 저장소에서 작업할 때 따라야 할 기본 규칙을 정의합니다.  
> 필요 시 팀 컨벤션에 맞게 수정하세요.

## 1) 목적
- 이 프로젝트에서 Codex의 코드 작성/수정 방식 일관성 유지
- 불필요한 리스크(파괴적 명령, 과한 변경, 무관한 리팩터링) 방지
- 리뷰 가능한 형태로 변경 내역과 검증 결과를 남기기

## 2) 작업 원칙
- 작은 단위로 변경하고, 변경 이유를 명확히 설명한다.
- 요청 범위를 벗어나는 리팩터링은 하지 않는다.
- 기존 코드 스타일과 구조를 우선 존중한다.
- 불확실한 요구사항은 추측 구현보다 질문을 우선한다.

## 3) 파일/코드 수정 규칙
- 가능한 한 **최소 수정(minimal diff)** 으로 해결한다.
- 하나의 작업에서 unrelated 파일은 수정하지 않는다.
- 자동 생성 파일/빌드 결과물은 사용자가 요청한 경우에만 커밋 대상으로 다룬다.
- 주석은 “무엇”보다 “왜”를 짧게 작성한다.
- 하드코딩 상수는 반복 사용 시 상수/설정으로 분리한다.

## 4) Git 안전 규칙
- 사용자 요청 없이 파괴적 명령(`git reset --hard`, `rm -rf` 등) 사용 금지
- 사용자 변경사항이 섞여 있으면 임의로 되돌리지 않는다.
- 커밋/브랜치 작업은 명시적으로 요청받은 경우에만 수행한다.

## 5) 명령 실행 규칙
- 파일 검색은 `rg` 우선 사용 (`rg --files`, `rg "pattern"`).
- 테스트/린트는 변경 범위에 맞춰 최소 단위부터 실행한다.
- 장시간/고비용 명령 실행 전 목적을 먼저 설명한다.

## 6) 검증 규칙
- 코드 변경 시 가능한 범위에서 아래를 수행한다.
  - 관련 테스트 실행
  - 린트/타입체크 확인
  - 실패 시 원인과 영향 범위 요약
- 검증 불가 환경이면 “무엇을 못 했는지”를 명확히 남긴다.

## 7) 응답 형식 규칙
- 최종 응답에는 아래를 포함한다.
  - 변경 파일 목록
  - 핵심 변경 사항 요약
  - 실행한 검증 명령과 결과
  - 남은 리스크/다음 권장 작업(있다면)

## 8) 코드 리뷰 기준(기본)
- 기능 정확성: 요구사항을 정확히 충족하는가
- 안정성: 예외/경계값 처리 누락이 없는가
- 가독성: 이름, 구조, 책임 분리가 명확한가
- 테스트 가능성: 변경을 검증할 수 있는가
- 회귀 위험: 기존 동작에 부작용이 없는가

## 9) 우선순위
1. 사용자의 직접 지시
2. 이 `AGENTS.md`
3. 저장소 내 기타 문서(`README`, `CONTRIBUTING` 등)
